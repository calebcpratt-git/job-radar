# Job Radar — App Expert

Job Radar is Caleb's personal job-search automation: a single Python script that runs every morning in GitHub Actions, pulls postings from named-company ATS boards plus two broad job-search APIs (JSearch and Adzuna), filters them against criteria in `config.yaml`, keeps everything posted in the last 30 days, and commits a static HTML dashboard served by GitHub Pages.

**Purpose and goals.** It exists so the user (currently searching for Product Operations roles in NYC, 1–3 yrs experience) sees a fresh, filtered brief of relevant postings without visiting job boards manually. Design goals: zero paid services (free-tier Adzuna and JSearch keys; every ATS endpoint below is a keyless public API/page), zero infrastructure (GitHub Actions + Pages do everything), one user-editable file (`config.yaml`), and resilience (one broken source never kills the run). Keep these in mind when proposing changes — solutions requiring servers, databases, or paid APIs cut against the app's philosophy.

**Architecture history (important):** the app used to track named company ATS boards and flag postings that were "new since last run" via a committed `seen_jobs.json` state file. That entire model is gone — there is no state file and no new-flagging. The app shows a **rolling 30-day window** of everything currently matching. Named-company tracking was later re-wired in a CSV-driven form: `ats_fetch.py` reads `ats_map.csv` (company, ats platform, endpoint URL, optional `waas_slug`) and dispatches each row to the matching fetcher, covering seven ATS platforms (see "Per-source API details" below). `main()` calls `get_all_ats_jobs()` **before** the broad sources so ATS-board results win dedupe over aggregator duplicates. Toggle this on/off via `ats_map.enabled` in `config.yaml`; the earlier token-based `fetch_greenhouse`/`fetch_lever`/`fetch_ashby`/`fetch_companies`/`ATS_FETCHERS` functions (driven by a `companies:` config list) were removed as dead code once this superseded them — if you see a reference to that shape, it's stale.

## Data flow (main() in job_check.py)

1. **Load config** → `criteria` dict: `location`, `location_aliases`, `remote_ok`, `max_days_old`, `digital_signals`, `title_rules`.
2. **Fetch named-company ATS boards** (`get_all_ats_jobs()` from `ats_fetch.py`) if `ats_map.enabled` (default true): reads `ats_map.csv`, dispatches each row to the fetcher matching its `ats` column, normalizes into the common job schema, then tags every job `group = "target"`. Runs **before** the broad sources so ATS-board postings win dedupe over aggregator duplicates of the same job. See "Per-source API details" for the 7 platforms covered and their quirks. One bad row (dead board, API change) logs to stderr and is skipped — never kills the run.
3. **Fetch Adzuna** (`fetch_adzuna`) if `adzuna.enabled`, tagged `group = "other"`: paginated (`pages` × `results_per_page`), stops early on a short page. `what_or` (the server-side keyword filter) is **not set in config** — it's built each run from `title_rule_words(criteria["title_rules"])` (see below), so Adzuna's own search always covers the same universe of words `title_rules` filters for locally; adding a rule to `title_rules` widens Adzuna's query automatically. Passes `where` (location) and `max_days_old` server-side. Reads `ADZUNA_APP_ID` / `ADZUNA_APP_KEY` from env; if missing it **skips with a stderr warning rather than failing**. Ids look like `az:<id>`. Captures both `location` (display name, often just a neighborhood) and `location_area` (full area hierarchy) for location filtering.
4. **Fetch JSearch** (`fetch_jsearch`) if `jsearch.enabled`, tagged `group = "other"`: Google for Jobs aggregator (surfaces LinkedIn, Indeed, Built In, etc.) via `api.openwebninja.com/jsearch/search-v2` with `x-api-key: JSEARCH_API_KEY`. Loops over `jsearch.queries` (a plain list of natural-language phrases in config, e.g. `"product operations in new york"`) — **one API call per query**, regardless of `pages`: `num_pages` is a page *count* bundled into a single response, not a page number. Results are nested under `data.jobs`. Maps `max_days_old` to the coarse `date_posted` buckets (today/3days/week/month). Ids look like `js:<job_id>`; `source` is `JSearch (<publisher>)`. Missing key → stderr warning + skip; a single query failing (timeout/429/etc.) logs `! JSearch query '<query>' failed: ...` and just skips that query, not the whole fetch. **Do not build this as one query combining many terms with OR/parentheses** — see "Known operational realities": the field is natural-language-only, and boolean syntax degrades results rather than broadening them. `title_rule_words()` (used for Adzuna's `what_or`, see above) is a shared helper — union of every word across all `title_rules`, first-seen order — but it is deliberately *not* used to build the JSearch query for this reason.
5. **Filter** everything through `matches()` (semantics below). `group` rides along on the job dict untouched by filtering.
6. **Dedupe, two passes**: first by `id` (fallback `url`), first occurrence wins; then by normalized `(title, company)` pair to kill the same posting arriving from multiple aggregators. Fetch order = priority order (ATS boards → Adzuna → JSearch), so an ATS-board copy of a posting wins over an aggregator's copy of the same job — including its `group`, so a job found via both a target company's board and JSearch is correctly kept as "target".
7. **Sort** by `posted` descending (jobs with no date sort last).
8. **Render** into `TEMPLATE` via `.format()`, writing `docs/index.html`. `render()` splits the final job list by `group` into `target_jobs` (`group == "target"`) and `other_jobs` (everything else) and builds each tab's `<li>` rows separately via `render_rows()`. Header shows "{n} matches · last {max_days_old} days" for the combined total; each tab button shows its own count.

## Filtering semantics (matches()) — the usual source of "why isn't X showing?"

- **title_rules** (config): a list of rules; each rule is a list of words. A title passes if **any one rule** matches, and a rule matches when **all of its words** appear in the title as whole words (title is tokenized with `re.findall(r"[a-z]+")`, so punctuation/hyphens split words but "ProductOps" stays one token and fails). A **single-word rule** (e.g. `[implementation]`, `[bizops]`) is valid and matches on that one word alone — the code just does `all(w in words for w in rule_words)`, which is trivially true for a one-element list — so those rules are much broader than the two-word AND rules. Rules as of 2026-07-16 span seven groups (see `config.yaml` for the exact list, which is the source of truth): the original product/ops variants (`[product, operations]`, `[product, ops]`, `[platform, operations]`, `[business, operations]`, `[strategy, operations]`, `[revenue, operations]`, `[product, analyst]`); product strategy & analytics (`product`+`strategy`/`analytics`/`specialist`); deployment / forward-deployed (`deployment`+`strategist`/`manager`/`associate`/`specialist`/`lead`, plus `[forward, deployed]` and `[deployed, intelligence]`); implementation (`[implementation]`, single-word); customer success & post-sale (`customer`+`success`/`operations`, `onboarding`+`manager`/`specialist`, `engagement`+`manager`/`associate`); solutions non-engineering variants (`solutions`+`consultant`/`associate`/`specialist` — deliberately excludes Solutions Engineer/Architect); and generalist ops (`gtm`+`operations`/`ops`, `sales`+`operations`, `growth`+`operations`, `[bizops]` single-word, `[chief, staff]`). If `title_rules` is empty, falls back to hardcoded product + (operations|ops). There is **no salary filter** anymore — salary is display-only.
- **digital_signals** (config): if the job has a non-empty description, it must contain at least one signal substring (software, saas, platform, api, roadmap, agile, ux, etc.) — filters physical-product/CPG/logistics ops out. **Jobs with no description are kept.**
- **location** (`location_ok()`): a job's location text (`location + " " + location_area`, lowercased) is checked against `location_aliases` (config) — a whole-word/phrase match via regex (`\b(?:alias1|alias2|...)\b`), **not** a plain substring check, so a short alias like `"queens"` doesn't false-match inside an unrelated word like "Queensland". `location_aliases` defaults to `[location]` if unset in config; as configured it's `["new york", "new york city", "nyc", "manhattan", "brooklyn", "queens", "bronx", "staten island"]`, so any of those — not just the literal string "New York" — counts as an NYC match. This closes the old "Brooklyn, NY" / "Manhattan" borough gap (both now match directly via their own alias) without needing a spelled-out "New York" anywhere in the text.
  Behavior beyond the alias match splits by `group` (see "Data flow" step 2–4):
  - **Other Postings** (`group != "target"`, Adzuna/JSearch): unchanged — a job passes if it matches an alias, **or** `remote_ok: true` and `looks_remote(job_loc)` (any remote job, regardless of where else it's tied to).
  - **Target Companies** (`group == "target"`, ATS-board jobs): a job matching an alias **always** passes, even if also remote-flagged (e.g. "New York, NY (Remote)" — the NY office is an option, so it's kept, unlike Other Postings' logic this isn't gated by `remote_ok`). A job that does *not* match an alias but *is* remote-flagged passes only if it names no other specific place: `_words(job_loc) - _REMOTE_FILLER_WORDS` (filler = `{remote, us, usa, u, s, a, united, states, the, in, only}`) must be empty — so `"Remote"`, `"Remote (United States)"`, `"Remote - US"` pass (could plausibly be worked from the NY office), while `"Remote - India"`, `"Madrid (Remote)"`, `"UK - Remote (Remote)"` are excluded (tied to a different specific place). A job with neither an alias match nor a remote flag is excluded outright.
- **max_days_old** (30): checked **inside matches() for every source** (this changed — it used to be Adzuna-server-side only). Parsed with `datetime.fromisoformat` after `Z`→`+00:00`; **unparseable or missing dates are kept**.

Adzuna also applies its own server-side keyword layer (`what_or`, derived from `title_rule_words(title_rules)` — see "Data flow" step 3, not a config value anymore) before the local filters — an Adzuna posting must survive both.

## Known operational realities

- **The dashboard is ~100% JSearch-dependent.** Adzuna typically contributes 0–2 jobs/day; JSearch supplies essentially everything. Each JSearch query is capped at whatever `pages` is (~10 results/page by Google relevance) means real matches can simply never enter the pipeline — when a "direct match" is missing, check source coverage *first*, filters second.
- **Sources fail soft.** A dead API key, rate limit, or API change produces a stderr warning, an empty result set, and a **successfully committed 0-match dashboard with a green Actions run**. Diagnose via the "Run job check" step log: look for `! JSearch query ... failed:` / `! Adzuna page N failed:` vs `· JSearch broad search: 0 postings`. (A patch adding a `raw == []` guard that exits 1 and injects a stale-data banner into the existing dashboard, plus `if: always()` on the commit step, may have been applied — check `main()` for `mark_dashboard_stale`.)
- **JSearch's `query` field is natural-language-only, not boolean** (confirmed 2026-07-16 against OpenWeb Ninja's own docs and live A/B tests): `"(product operations OR chief of staff) in New York"` returns *fewer*, less relevant results than either phrase alone — the API doesn't parse OR/parentheses as logic, it just gets confused. Broadening JSearch coverage means adding more short, separate, plain-language entries to `jsearch.queries` (currently 4: `"product operations in new york"`, `"operations manager in new york"`, `"forward deployed implementation manager in new york"`, `"customer success solutions consultant in new york"`) — never combining terms into one query. Bumping `pages` does not compensate for a narrow/malformed query; results per query saturate independently of page count once the underlying search has nothing more to return (tested 10 vs 20 vs 30 vs 50, no gain beyond a query's own saturation point).
- **JSearch is noticeably flaky under load.** Testing on 2026-07-16 (repeated runs of the same 4 queries within a short span) produced inconsistent per-query failures — some client-side read timeouts (20s `HTTP_TIMEOUT`), some upstream `504 Endpoint request timed out`, and eventually explicit `429` rate-limit responses after enough back-to-back calls. Higher `pages` per query makes individual calls slower and more likely to time out (at `pages: 30`, 3 of 4 queries failed vs 1 of 4 at `pages: 10`). Treat a partial/empty JSearch result as possibly transient, not necessarily a broken key or query — retrying later (not immediately, to avoid compounding rate-limiting) often succeeds.
- **JSearch free tier** is ~200 calls/month; call cost scales with number of queries (1 call/day/query), so with 4 queries configured that's ~120–150 calls/month at once/day — keep the `queries:` list short and be mindful that manual/local test runs consume the same quota as the scheduled workflow.
- **ATS boards contribute little under the current title_rules** — though this note predates the 2026-07-16 title_rules expansion. `ats_map.csv` is mostly AI/fintech/proptech startups; their listings skew Engineering/Sales/GTM, not classic Product Operations, so historically only a handful of ~3,000 fetched ATS postings survived `title_rules`. The expanded rules now include customer success, onboarding, solutions, implementation, and forward-deployed titles — role types these startups commonly post — so ATS-board contribution may rise; hasn't been re-measured against a live run yet.

## Deployment & operations

- **Secrets** (GitHub repo Actions secrets → env vars): `ADZUNA_APP_ID`, `ADZUNA_APP_KEY`, `JSEARCH_API_KEY`. Never hardcode. Local runs can use a `.env` file (loaded by `_load_dotenv`, never overrides real env vars).
- **Workflow**: checkout → Python 3.12 → `pip install -r requirements.txt` → `python job_check.py` → commit `docs/index.html` as `job-radar-bot` (`|| echo "No changes"` prevents failures on empty commits). Requires `permissions: contents: write`.
- **Pages**: deploy-from-branch, `main` / `/docs`. Dashboard URL: `https://calebcpratt-git.github.io/job-radar/`.
- **Local run**: set the three env vars (or `.env`), `python job_check.py`, open `docs/index.html`. No state file, so local runs are harmless — just don't commit the locally generated dashboard.

## Dashboard behavior

Self-contained generated HTML (Google Fonts only external dependency): header with "{n} matches · last 30 days", then a two-column `.layout` — a `.sidebar` (the Companies filter) and `.main` (tabs, controls, job lists, footer) — inside the `.wrap` container (widened from 760px to 980px to fit the sidebar; below 760px viewport width the layout stacks to a single column). **Two tabs** — "Target Companies" (postings with `group == "target"`, i.e. from `ats_map.csv` boards) and "Other Postings" (everything else — Adzuna + JSearch broad search) — each with its own count badge in the tab button and its own `<ul class="job-list">` of `<li class="job">` rows (title link, company, location, salary if known, source · posted-date meta line). Only one tab's `<ul>` is visible at a time (`hidden` attribute toggled by a small inline `<script>`); the other stays in the DOM. Each tab has its own empty state ("No target-company matches today…" / "No other matches today…"). Salary formatting via `fmt_salary`, relative dates via `fmt_posted` (both driven by the shared `days_old()` helper — whole days since `posted`, or `None` if missing/unparseable). Colors/typography are CSS variables at the top of `TEMPLATE`. Because `TEMPLATE` is filled with `str.format`, **literal braces in any CSS/JS added there must be doubled (`{{ }}`)** — forgetting this raises `KeyError`/`IndexError` at render time. Job fields are escaped with `html.escape`. There is no "new" pill or new-only toggle anymore.

A job's tab placement is decided once, at fetch time, by the `group` field set in `main()` (`"target"` for ATS-board jobs, `"other"` for Adzuna/JSearch) — see "Data flow" step 2–4 — and `render()` splits on it before building each tab's rows. To add a third tab or re-source a tab, change what sets `group` in `main()`, not `render()`.

**Client-side filters (all in the inline `<script>`, purely front-end — no fetch/config involvement):** every `<li class="job">` carries `data-company` (company name, trimmed + lowercased, the matching key) and `data-posted-date` (UTC calendar date `YYYY-MM-DD` from `posted_date_str()`, or `""` if `posted` was missing/unparseable — treated as "always shown," mirroring `matches()`'s own handling of unparseable dates). Three controls share one `filterState` object (`{{search, companies, sinceDate}}`) and a single `computeVisible(el)` predicate that ANDs all three together; `applyFilters()` re-evaluates every job's `style.display` whenever any control changes. This replaced the old standalone search-only `apply()` function — the search box (`#q`) behavior is unchanged (still filters `.job` items by substring across **both** tabs, independent of which tab is shown), it just now writes into `filterState.search` instead of setting `display` directly.
- **Posted since** (`#date-filter`, a native `<input type="date">` in `.controls` next to the search box, with a "Posted since" label and `max` pinned to today's UTC date so a future date can't be picked): picking a date sets `filterState.sinceDate` to that `YYYY-MM-DD` string; a job is hidden when its `data-posted-date` is a non-empty string earlier than `sinceDate` (plain ISO-date string comparison — no `Date` parsing). `data-posted-date=""` always passes. Clearing the input (browsers show a native "×" once a date is set) sets `filterState.sinceDate` back to `null`, showing everything again. `posted_date_str()` (near `days_old()`/`fmt_posted()` in `job_check.py`) is the Python-side source of `data-posted-date`, converting a job's `posted` ISO timestamp to its UTC calendar date.
- **Companies** (`.company-filter` in `.sidebar`): a checkbox per distinct company, scoped to whichever tab is currently active — `buildCompanyFilter(tabKey)` rebuilds the checkbox list from only that tab panel's `.job` elements' `data-company` (deduped by that key, labeled with the original-case company name read from `.company`'s text, not derived from the lowercased key, sorted alphabetically), so a company that only has postings on the other tab is never offered. Called once on load for the default tab (`'target'`) and again on every tab-button click, right after the `hidden` toggle — it also resets `filterState.companies` to a fresh empty `Set` each time (a prior tab's checked companies don't carry over, since they may not exist on the new tab) and calls `applyFilters()` itself. The list box is capped at `max-height:310px` with `overflow-y:auto` on `.company-filter-list` — sized for ~10 rows at the current row padding/font-size before it scrolls. Zero boxes checked = no company filtering (shows all); checking one or more shows the union of matching companies (`filterState.companies` is a `Set`, checked via `.size > 0`).
- Adding a fourth filter: follow the same pattern — add a field to `filterState`, a branch in `computeVisible()`, and a control that updates state then calls `applyFilters()`. Don't reintroduce direct `style.display` writes outside `applyFilters()` or a new filter will silently fight with the existing ones.

## Common tasks — how to do them right

- **Change search criteria**: edit `config.yaml` only. A filter can only keep what the sources return, so broadening usually means touching both a local layer (`title_rules` / `digital_signals`) and what the sources actually fetch — but the two source queries behave differently now: Adzuna's `what_or` is auto-derived from `title_rules` (nothing to edit there directly), while JSearch's `query` field is natural-language-only and ignores `title_rules` entirely, so widening JSearch coverage means adding another short, plain-language entry to `jsearch.queries` in config — never combining terms into one query with OR/parentheses (see "Known operational realities").
- **Add a title pattern**: append a rule to `title_rules` (all words in a rule are ANDed; rules are ORed). Whole-word matching — add singular/abbreviated variants as separate rules if needed.
- **Track a specific company**: add a row to `ats_map.csv` (`company, ats, endpoint, open_jobs, waas_slug`). `ats` must be one of `ashby` / `workable` / `greenhouse` / `lever` / `rippling` / `gem`, with `endpoint` the full API/board URL (see patterns below); or leave `ats` as `NOT FOUND` and `endpoint` empty with `waas_slug` filled in for a Work at a Startup (YC) company. Verify a new endpoint by requesting it directly before trusting it — platforms occasionally change response shape without notice. A company on an unsupported ATS or a self-hosted careers page needs a new fetcher in `ats_fetch.py` (raw fetch + normalizer returning the common job-dict shape with a unique id prefix) wired into `FETCHERS`/`NORMALIZERS` and `get_all_ats_jobs()`'s dispatch, with failures caught same as the rest.
- **Change schedule**: edit the cron in `.github/workflows/job-check.yml` (UTC!), not the root duplicate.
- **Widen/narrow the window**: `max_days_old` in config — applies to all sources in `matches()` and maps to JSearch's coarse `date_posted` buckets.
- **Restyle the dashboard**: edit `TEMPLATE` in `job_check.py` (double the braces), rerun, optionally sync the root `index.html` preview.
- **Debug a missing posting, in order**: (1) is it in the raw source results at all? — JSearch coverage is the top cause of misses; (2) title_rules whole-word match; (3) `location_ok()` — does the location text match a `location_aliases` entry, or (for a `group == "target"` posting) is it remote-flagged but tied to a different specific place per `_REMOTE_FILLER_WORDS` (see "Filtering semantics"); (4) digital_signals if it has a description; (5) 30-day window / unparseable date.

## Per-source API details (ats_fetch.py — 7 platforms via ats_map.csv)

All requests use a full browser-style User-Agent + `Accept` header (`DEFAULT_HEADERS` in `ats_fetch.py`) — workatastartup.com specifically 406s on a bare `Mozilla/5.0` with no `Accept` header, so don't simplify this. Ids are namespaced `<prefix>:<company-slug>:<platform-id>` (`_slug_from_endpoint` backs up one path segment when the URL ends in a literal `/jobs`, e.g. Greenhouse/Rippling).

- **Ashby** (`ats: ashby`) — `endpoint` = `api.ashbyhq.com/posting-api/job-board/{slug}`, fetched with `includeCompensation=true`. Jobs at `data.jobs[]`. Appends "(Remote)" to location when `isRemote`; extracts salary from nested `compensationTiers`; description from `descriptionPlain`.
- **Workable** (`ats: workable`) — `endpoint` = `apply.workable.com/api/v1/widget/accounts/{slug}`. Jobs at `data.jobs[]`. Location built from `city`/`state`/`country` + "(Remote)" if `telecommuting`. `published_on` is date-only (`YYYY-MM-DD`) — normalized to a UTC datetime so `max_days_old` filtering actually works; a raw date-only string fails `datetime.fromisoformat` arithmetic silently (caught and treated as "unparseable, keep"). **No description field** in this endpoint — such jobs bypass `digital_signals`. No salary data.
- **Greenhouse** (`ats: greenhouse`) — `endpoint` = `boards-api.greenhouse.io/v1/boards/{slug}/jobs`, fetched with `content=true` for the HTML description (stripped to plain text). `posted` is really `updated_at` — an edited old posting looks recent and can re-enter the 30-day window. No salary data.
- **Lever** (`ats: lever`) — `endpoint` = `api.lever.co/v0/postings/{slug}?mode=json` (query string included in the CSV row). Bare list, not `{"jobs": [...]}`. `posted` from `createdAt` epoch-ms; location from `categories.location`; salary from `salaryRange.min`/`max`; description from `descriptionPlain`.
- **Rippling** (`ats: rippling`) — `endpoint` = `api.rippling.com/platform/api/ats/v1/board/{slug}/jobs`. **Bare list**, not `{"jobs": [...]}` — a naive `.get("jobs", ...)` on the response crashes with `AttributeError` since lists have no `.get()`; `fetch_rippling` checks `isinstance(data, list)` first. Location from `workLocation.label`. No date, no salary, no description exposed.
- **Gem** (`ats: gem`) — `endpoint` = `jobs.gem.com/{slug}`, but that URL is a client-rendered GraphQL SPA with **no embedded JSON and no REST endpoint**; there is nothing to GET directly. `fetch_gem` instead POSTs a `JobBoardList` GraphQL query (reverse-engineered from Gem's own JS bundle) to `jobs.gem.com/api/public/graphql` with `{"variables": {"boardId": slug}}`. Jobs at `data.oatsExternalJobPostings.jobPostings[]`. Job URL is constructed as `jobs.gem.com/{slug}/{extId}`. **No posted-date or salary in this query** (only Gem's single-job-page query has those, which would mean one extra request per job — not fetched). Breaks if Gem changes this internal query; there is no public docs page to check against.
- **Work at a Startup / WaaS** (`ats: NOT FOUND`, `waas_slug` filled in) — **not** the Algolia search embedded on workatastartup.com: that key's baked-in `tagFilters` restriction returns zero hits for every query, including an empty one (confirmed by direct testing, despite Algolia "looking" like the obvious integration point). Instead `fetch_waas(slug)` fetches `workatastartup.com/companies/{slug}` and parses the server-rendered Inertia.js `data-page` JSON attribute for `props.company.jobs[]` — the company's own page already lists its open jobs, keyed 1:1 to `waas_slug`, no API key needed. Salary is a display string (`"$20K - $40K"`), parsed into `salary_min`/`salary_max` via regex. Job URL is `workatastartup.com/jobs/{id}`. **No posted-date or description exposed.** Breaks if workatastartup.com changes its Inertia component/props shape (no versioned public API).
- Rows with `ats: NOT FOUND` and no `waas_slug` (no usable source at all) are skipped with a stderr warning, not treated as an error.
